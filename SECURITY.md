# COLOSSO — заметка по безопасности (RLS-аудит)

Приложение — браузерный SPA с **publishable (anon)** ключом Supabase прямо в коде. Это нормально: вся защита держится на **Row Level Security (RLS)**. Ниже — что нашёл аудит и что закрыто в `security-hardening.sql` (запускать последним).

## Закрыто (HIGH)
1. **Подмена результата лобби.** Раньше любой залогиненный пользователь мог редактировать любое лобби (баны, победитель, статус) — и накручивать рейтинг. Теперь читать/писать лобби могут только две стороны скрима.
2. **Само-начисление GLR/XP.** `finish_match()` не проверял участие — можно было звать его бесконечно и фармить рейтинг. Теперь функция требует, чтобы вызывающий был участником матча, и идемпотентна (завершённый матч нельзя начислить повторно).
3. **Двойное начисление.** Награда шла и из `finish_match`, и из триггера `trg_lobby_done`. Триггер убран — единственный источник начисления теперь `finish_match`.

## Закрыто (MEDIUM/LOW)
4. **Спуф уведомлений.** `notifications` теперь принимает от клиента только `invite`/`application`; системные/матчевые шлёт только серверная функция (SECURITY DEFINER, минует RLS).
5. **Фейковая история матчей.** Прямой клиентский INSERT в `matches` убран — историю пишет только `finish_match`.
6. **Само-добавление в команду.** В `team_members` теперь нельзя вступить без принятой заявки (или будучи владельцем).
7. **Аватары.** Загрузка ограничена папкой пользователя (`<uid>/…`), добавлено право удалять свои файлы.
8. Добавлены недостающие DELETE-политики (свои заявки/турниры/пространства).

## Анти-сговор / целостность рейтинга (`anti-collusion.sql`)

Проблема: исход матча сообщает игрок, а сговор **двух сторон** (оба шлют один и тот же фейк) не ловится никакой взаимной сверкой скринов. Поэтому защита — **структурная**, на уровне дизайна, а не верификации матча. Три механизма:

1. **Ранкед-гейтинг.** GLR начисляется **только за матчи с назначенным соперником** (матчмейкинг / турнир). Скрим — это самостоятельно выбранный соперник, значит можно позвать команду-сообщника, поэтому скримы **рейтинг-нейтральны** (скрин только «для истории»). Технически: у `lobbies` появились `ranked` и `source`; матчмейкинг/турнир обязаны создавать лобби с `ranked = true`. Скрим-лобби остаются `ranked = false` по умолчанию.
2. **Zero-sum Elo.** `finish_match` теперь делает настоящий Elo-расчёт: победитель забирает ровно столько, сколько теряет проигравший. Накрутка кому-то стоит реального рейтинга другого аккаунта — «бесплатного» сговора нет. Матч сводится **атомарно при первом вызове участника**, дальше функция идемпотентна (повторный вызов просто вернёт текущий стендинг).
3. **Затухание + дневной лимит.** GLR за матч против **того же** соперника затухает с каждым повтором в окне 14 дней (1, ½, ⅓, ¼, дальше 0), а дневной кап (`DAILY_GAIN_CAP`) обнуляет **прирост** сверх лимита (проигрыши при этом считаются всегда — быть «фарм-партнёром» по-прежнему больно). Пропускная способность накрутки падает к нулю.

Поверх — **детект аномалий** (`scan_anomalies()` + таблица `anti_cheat_flags`, запускать по расписанию через pg_cron): повторяющиеся пары, односторонний переток рейтинга (бустинг), свежесозданные аккаунты, играющие друг друга, ночные всплески. Аналитика закрыта от клиента (только service-role).

Все ручки тюнятся константами в начале `finish_match` / `_settle_lobby` (ищи `TUNABLE`).

## Репорт результата + диспуты (`result-reporting.sql`)

Закрывает вопрос «кто реально выиграл?» без доверия одному самоотчёту. Логика расчёта вынесена в единый `_settle_lobby()` (его переиспользуют и `finish_match`, и репорт).

- **Двусторонний репорт.** `report_result(lobby, won, score, screenshot)` — каждая сторона сообщает свой исход. Совпали → авто-зачёт через `_settle_lobby` (настоящий Elo). Разошлись → лобби помечается `disputed`, создаётся строка в `disputes`, обе стороны получают уведомление.
- **Разбор админом.** `resolve_dispute(lobby, side)` — только `is_admin()`. После решения триггер снимает trust у того, кто сообщил ложный результат.
- RLS: писать репорты можно только через `report_result` (SECURITY DEFINER), читать — только участникам матча.

## Турниры (`tournaments-glr.sql`)

`report_tournament_match(t, round, match, winner_team, scoreA, scoreB)` — только организатор. Zero-sum Elo между **командами**, начисление GLR/XP каждому игроку обеих команд, продвижение победителя по сетке (`bracket` jsonb), статус `finished` на финале. Идемпотентно через `tournament_results`. Здесь источник истины — оператор (он видит лобби), а сетка рандомит соперников, поэтому организаторский репорт уместен.

## Операционка (`ops-hardening.sql`)

- **Rate-limit** (BEFORE INSERT): скримы 10/мин, отклики 20/мин, заявки 10/мин, сообщения 30/мин. Серверные (service-role) вставки не блокируются.
- **Контент-фильтр** ников и чата по таблице `blocked_terms` (админ расширяет).
- **Trust-штраф** за ложный репорт в разрешённом диспуте.
- **Анти-чит скан** на pg_cron + `list_flags()` для админ-разбора `anti_cheat_flags`.

## Прочее
- `lobby_ban_map` (`lobby-ban-map.sql`) — недостающая RPC бана карт, теперь реализована (участник-гейтед, авто-decider).
- Регресс-тесты: `backend/test/` — `npm install && npm test` поднимает настоящий Postgres (PGlite) и гоняет всю цепочку (42 проверки).

## Остаточные риски / на будущее
- **Доверенный источник на крупных призовых.** Репорт+диспут закрывают массовый случай; на больших деньгах финал всё равно стоит спектейтить/стримить.
- **Антифрод по паттернам** можно усиливать (сигналов в `scan_anomalies` сейчас 4).
- **Выплаты** (Pix/Mercado Pago) — отдельный комплаенс, выключены.

## Порядок применения SQL
(именно в этом порядке прогоняются регресс-тесты)

`schema.sql` → `glr-xp.sql` → `matchmaking.sql` → `bracket.sql` → `matches.sql` → `security-hardening.sql` → `anti-collusion.sql` → `result-reporting.sql` → `tournaments-glr.sql` → `lobby-ban-map.sql` → **`ops-hardening.sql`** (последним).
